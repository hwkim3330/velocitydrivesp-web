Configurations in VelocityDRIVE™ CT

Introduction

As shown in See User Interface, VelocityDRIVE CT can monitor connected switches in the See Devices Tab and build YANG catalog changes in the See Configurations Tab. It is best to think of the See Devices Tab (and the explorable YANG libraries) as being the active state of a switch. The See Configurations Tab should be viewed as a staging area for planned switch changes. Using both tab functions in tandem will allow a user to:

  • See the switch's current state in action
  • Copy the active Device state parameters to a Configuration and make changes
  • Upload the new Configuration to the Device
  • Confirm if those changes take affect on the Device(s) in real time

In this chapter we will outline the different levels of Configurations a user can create, the various methods used in creating Configurations, and how Configurations are applied to a device. Configuration Types

Configuration Types

There are two main configuration types that a user can create. This first is a See Full Datastore Configuration based on a Device's entire YANG Catalog. The second is a See Patch Datastore Configuration, which applies changes to only a subset of the YANG Catalog structure. If this is your first time interacting with the VelocityDRIVE ™ Configuration Tool User's Guide, we recommend using the following steps first to create a Full Datastore Configuration to fully understand the operating principles of VelocityDRIVE CT.

Creating Configurations

Full Datastore Configurations

The simplest way to change the default values of a device’s running datastore is to retrieve the switch’s entire YANG instance for editing in a Full Datastore Configuration. VelocityDRIVE CT configurations are stored offline and do not become active on the switch until temporarily synced or uploaded and saved.

Note:

Once the device’s default configuration is retrieved by VelocityDRIVE CT, it recommended to export and save it as a reference. See Chapter 7. “Default Configuration Backup Advisory” for more details.

Once the device's default Configuration is copied by VelocityDRIVE CT, we recommend that it is exported saved as a reference. See See Default Configuration Backup Advisory for more details.

Patch Datastore Configuration

For instances when only a particular segment of a switch's YANG Catalog will be modified, a Patch configuration can be created. Patch Datastore Configurations are smaller in file size and easier to navigate, as they only contain the segment of the YANG Catalog tree that is being changed. To create a Patch, we will create a new configuration in the Configurations tab.

Creating a New Patch Configuration

Figure 4-2: Creating a New Patch Configuration

  1. Navigate to the Configurations tab
  2. Click the Add New Configuration button
  3. A new Configuration card will appear

After creating any new configuration, we recommend giving the configuration a unique name, as described in See Naming VelocityDRIVE CT Configurations. For this example, we will use the name “Patch”.

Selecting Patch YANG Tree Segment

Now that the empty Patch Configuration has been created, navigate to the Device tab, and select the Explore button of the Device you will be working with.

  2. In the lower left Configurations dashboard, press the Copy Datastore button in the newly created “Patch” configuration file.
  3. Clicking the Open Configuration button will open the Patch configuration for editing (see Figure 4-4).

Alternate Patch Creation Methods

Patches can also be created in the following ways.

Alternate Method 1

Create a full datastore configuration (as described in Section 4.3.1 “Full Datastore Configuration”). Then, selectively remove all unneeded branches (see Section 4.4 “Configuration Editing” for instructions on removing branches). Any changes made to the remaining branches is considered a patch.

Alternate Method 2

Export a full datastore of your YANG Catalog from the /(root) of the Configuration tab using the Export button , saving it as a *.JSON file. Using a text editor, remove all unnecessary branches and make modifications to the isolated branch area. Then using the Import button , upload the Patch (setting upload type to “Patch,” not “Replacement”) to the appropriate branch segment of your existing Configuration. For more information on importing and exporting, please see Section 4.8 “Export And Import of Device Library and Configurations”.

Note: Importing a Patch to a Configuration at the incorrect branch/node level will result in an error.

Configuration Editing

In VelocityDRIVE CT, navigating through an active device’s YANG tree structure is a read-only experience. But when exploring a datastore retrieved from a device for use in a configuration, the majority of leaf values and variables are now editable.

Applying a Configuration

Figure 4-12: Applying a Configuration

  1. Continuing with the example in Section 4.4.5 “Editing Values in a Configuration”, we have edited our values to “non-revertive” and “1” in the Value column
  2. Pressing the Apply Configuration button , adjacent to the target Device in the Device dashboard, will apply these changes temporarily to the Device.

The benefit of using the Apply Configuration button is that it empowers the user to observe how different values affect a device in real time. Values can be tweaked and applied until the desired result is achieved.

Note: If the values are not what you expect them to be (after either using the Apply Configuration button or after a reset/power cycle), use the Refresh button while in the Device tab to send a FETCH command to the device (see Section 3.7.2 “Devices Tab Interface” for more details).

To permanently apply changes to a device, a Full Datastore (root level) or Patch (branch level) Configuration will need to be uploaded (see Section 4.6 “Configuration Upload” for more details)

Configuration Upload

Configurations can fall into two types, Full Datastore or Patch. Please see Section 4.2 “Configuration Types” for more information. VelocityDRIVE CT applies Configuration changes as new start up configurations, bypassing the process of exporting and uploading binary files.

Note: Full Datastore configurations are applied using the CORECONF GET command, while Patches are executed using the iPATCH command.

Figure 4-14: Configuration Upload

Once you have made your parameter changes to your new Configuration (whether either a Full Datastore or Patch), navigate to the root node of the Configuration tab. In the Devices dashboard, click the Upload button as seen circled in Figure 4-14.

Note: After uploading the start up Configuration, you must lightly press your device’s reset button just once, as instructed by the notification message shown in Figure 4-15. Do not perform a power cycle.

Figure 4-15: Configuration Upload Confirmation Message

Once the Device completes boot, the changes from the Configuration will remain on the device. Configuration changes are persistent unless a subsequent Configuration upload alters the parameters, or firmware is flashed using a secondary tool.

troubleshooting Configurations with Delete Toggle

If a Configuration is not behaving as expected, you may want to troubleshoot the design by seeing how the device behaves with particular containers no longer present. Yet removing and then reincorporating branches (and their values) one by one can be a tedious process (and may introduce errors). This is when the Delete Toggle feature is extremely useful.

Figure 4-16: Delete Toggle

Located in top left of the user interface main panel in certain types of elements is the Delete Toggle button. Sliding it into the “on” position effectively deactivates or “mutes” the element (and all nodes and leafs nested under it) from the Configuration during the next upload or sync process.

Note: With this toggle activated, the tool will perform the next upload to the device as an iPatch command, replacing the container with a null instance identifier.

Figure 4-17: Delete Toggle Activated

In Figure 4-17, we can see that the device was updated successfully with the container suppressed (notice the presence of the red stop icon now adjacent to the container). The container can be reinstated to the Configuration by deactivating the Delete Toggle and re-applying the Configuration.

Export and Import of Device Library and Configurations

Device libraries and VelocityDRIVE CT Configurations can be exported and imported, allowing for backup and restoration of data outside of the VelocityDRIVE CT Project file format

Exports of both Device libraries and user created Configurations are saved in various text-based formats. This provides the added benefit of allowing users to edit exports in the text-based coding program of their choice. Exported files can also be diffed for debug comparison purposes. We advise users to make a backup of their Device’s original library as a regular best practice. Please see Chapter 7. “Default Configuration Backup Advisory” for more details.

Device State Export (CBOR, JSON)

The current state of your Device can be exported directly from VelocityDRIVE CT as a CBOR or JSON text file.

Figure 4-18: Device Configuration Export

The content of any container on the Device can be exported in various formats. This action can be performed at / (root) (for the full datastore) or at any lower level in the YANG tree (for partial backups or patches).To export the content, click the Export button (shown in Figure 4-18) and choose the file format extension. The following formats are available:

  • CBOR Diagnostic Notation
  • CBOR Binary Image
  • CBOR Binary Image as Hexadecimal Dump (text format to compare binary images with diff-tools)
  • JSON File, which represents the YANG tree of instances

Device State Import (CBOR, JSON)

A CBOR-based start-up Configuration can also be applied directly to a Device using the Upload button on Device card as shown in Figure 4-19.

Figure 4-19: Device Card Upload Button

Pressing the Upload button will bring up a dialog menu. Make sure that Upload Type is set to Startup Configuration and the CoAP Method is set to PUT. Next, browse for the file. Click the Upload button in the dialog. Check that the log displays CREATED in response to the upload. Briefly press the reset button to power cycle (be aware that pressing and holding the button will reset the startup configuration).

Configuration Export (YAML, JSON)

The content of a configuration can be exported in various formats depending on the level of the YANG tree that the export originates from. Export the content using the Download button to the right of the container level.

Figure 4-20: Device Configuration Export

As shown in Figure 4-20, performing this action at the absolute / (root) level of a configuration will generate an export in the following file formats:

  • iPATCH YAML File (*.patch.yaml), for exports of configurations made of specific containers/branches
  • FETCH YAML File (*.fetch.yaml), for exports of full datastores
  • iPATCH JSON File (*.patch.json), for exports of configurations made of specific containers/branches
  • FETCH JSON File (*.fetch.json), for exports of full datastores

Exports performed at any level other than / (root) will generate file exports in the following formats:

  • YAML File (*.yaml)
  • JSON File (*.json), which represents the YANG tree of instances
  • CBOR Diagnostic Notation File (*.cbor.diag.json)
  • CBOR Binary File (*.cbor.bin)
  • CBOR Text File (*.cbor.txt), a text format to compare binary images with diff-tools

Configuration Import (YAML, JSON)

Device Configuration Import

Uploading a Configuration into VelocityDRIVE CT requires that the user is aware of several factors so that the correct upload method is selected. Pressing the Upload button to the right of the container will display the Import menu. Choose the Merge Type for the appropriate circumstance:

  • Patch - Select if uploading a small change to a previously existing Configuration file.
  • Put - Select if uploading a Full Datastore or Patch to an empty Configuration file.

Note: Importing a Patch to a Configuration at the incorrect branch/node level will result in an error
